# Аудит Windows Fix Toolkit (MVP)

Дата: 2026-03-05

## Резюме

Проведён статический аудит кода и сценариев запуска. Найдены критичные и значимые проблемы в обработке exit code, надёжности запуска внешних процессов, логировании и UX dry-run.

## Ключевые проблемы

## 1) Критическая: неверный итоговый ExitCode для режимов Repair/Full
**Симптом:** если шаг `Repair: DISM CheckHealth` упадёт, toolkit в большинстве случаев всё равно вернёт `0`.

**Где:** `Invoke-WindowsFix`, финальная логика возврата кодов.  
Сейчас `return 1` происходит только для шагов, имя которых матчится на `DISM RestoreHealth|SFC` или для SelfTest с любым FAIL.

**Риск:** CI/автоматизация ошибочно считает запуск успешным при реальной ошибке repair.

**Решение:**
- Ввести список критичных шагов для каждого режима и возвращать `1`, если любой из них `FAIL`.
- Для текущего MVP минимум считать `Repair: DISM CheckHealth` критичным в `Repair/Full`.

---

## 2) Критическая: риск deadlock при чтении StdOut/StdErr
**Симптом:** `Invoke-ExternalCommand` делает синхронные `ReadToEnd()` для stdout и stderr до/вместо корректного асинхронного дренажа потоков.

**Где:** `src/internal/process.ps1`.

**Риск:** процесс может зависнуть при большом выводе (заполнении буфера одного из потоков).

**Решение:**
- Использовать `BeginOutputReadLine`/`BeginErrorReadLine` + события, либо параллельное чтение потоков.
- После `WaitForExit(timeout)` делать дополнительный `WaitForExit()` для завершения асинхронной обработки.

---

## 3) Высокая: аргументы процесса фактически не экранируются
**Симптом:** в `ProcessStartInfo.Arguments` передаётся `($argsClean -join ' ')`, а не корректно экранированный набор аргументов.

**Где:** `src/internal/process.ps1`.

**Риск:** параметры с пробелами/кавычками могут ломать запуск или менять семантику команд.

**Решение:**
- На .NET с PowerShell 7 использовать `ProcessStartInfo.ArgumentList`.
- Для 5.1 реализовать безопасную функцию escape для Win32 command line и использовать её для фактического запуска (не только для логирования).

---

## 4) Высокая: DryRun требует admin (неожиданное поведение)
**Симптом:** `DryRun` мапится на `Full`, затем срабатывает admin-check для Repair/Full.

**Где:** `src/WindowsFixToolkit.psm1`.

**Риск:** пользователь не может получить план действий без админ-прав, хотя в dry-run изменений не делается.

**Решение:**
- Пропускать admin-check при `Mode = DryRun`.
- Формировать план шагов как `SKIPPED/PLANNED` независимо от привилегий.

---

## 5) Средняя: файл отчёта не создаётся при ранних ошибках пути/модуля
**Симптом:** часть ошибок в entrypoint (до вызова `Invoke-WindowsFix`) не попадает в report.json/report.md.

**Где:** `bin/windowsfix.ps1`.

**Риск:** теряется диагностика «почему не стартовало».

**Решение:**
- Обернуть `Import-Module` и bootstrap в `try/catch` с аварийным `bootstrap-error.log`.
- Возвращать код `3` при bootstrap исключениях с понятным сообщением.

---

## 6) Средняя: логгер ретраит только lock-тексты на английском
**Симптом:** retry привязан к подстрокам `used/being used by another process`.

**Где:** `src/internal/logging.ps1`.

**Риск:** на локализованных ОС lock-ошибка может не ретраиться.

**Решение:**
- Ретраить по типу исключения (`IOException`) и HResult, а не по тексту.
- Сохранить fallback по тексту как дополнительный путь.

---

## 7) Средняя: DIAGNOSE неполный относительно заявленного функционала
**Симптом:** отсутствуют HTTPS HEAD check и event log анализ, заявленные в требованиях/описании.

**Где:** `Invoke-Diagnose`.

**Риск:** пропуск типичных причин проблем (сеть/TLS, ошибки приложений/CBS/DISM/WER).

**Решение:**
- Добавить `Invoke-WebRequest -Method Head` с timeout.
- Добавить чтение последних N событий по каналам Application/System + фильтрам источников.

---

## 8) Средняя: smoke test слишком поверхностный
**Симптом:** тест проверяет только экспорт функций, но не поведение режимов/exit-codes.

**Где:** `tests/smoke.tests.ps1`.

**Риск:** регрессии в логике выполнения не ловятся.

**Решение:**
- Добавить минимум:
  - `Invoke-WindowsFix -Mode SelfTest` в мок-режиме/с подменой раннера,
  - проверку exit code `2` при Repair без админа,
  - проверку генерации `report.json`.

---

## 9) Низкая: launcher и git-monitor не журналируют свои действия в файл
**Симптом:** только консоль, без истории изменений/команд.

**Где:** `bin/windowsfix-menu.sh`, `bin/project-git-monitor.sh`.

**Риск:** сложно восстанавливать операционный контекст.

**Решение:**
- Добавить опциональный `--log-file` и запись ключевых действий.

---

## 10) Низкая: README утверждает функциональность шире фактической
**Симптом:** часть требований из постановки пока не реализована (event logs, HTTPS head, полноценные repair-steps).

**Риск:** завышенные ожидания у пользователя.

**Решение:**
- Явно пометить `Implemented` vs `Planned` по режимам/шагам.
